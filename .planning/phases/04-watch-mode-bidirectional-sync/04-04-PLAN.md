  <behavior>
    - Test 1: When running flag is set to false, the watch loop exits within 1 second (no filesystem event needed to unblock)
    - Test 2: Normal filesystem events are still processed when running is true
  </behavior>
  <action>
In `WatchEngine::watch()`, replace the blocking `for events in rx` iterator (line 67) with a loop that uses `rx.recv_timeout(Duration::from_millis(500))`. The new loop structure:

```rust
loop {
    if !running.load(SeqCst) {
        break;
    }

    let events = match rx.recv_timeout(Duration::from_millis(500)) {
        Ok(events) => events,
        Err(std::sync::mpsc::RecvTimeoutError::Timeout) => continue,
        Err(std::sync::mpsc::RecvTimeoutError::Disconnected) => break,
    };

    // ... rest of event handling unchanged ...
}
```

This ensures the `running` flag is checked every 500ms regardless of whether filesystem events arrive. The existing event handling logic inside the loop (syncing check, canonical vs tool-native detection, forward/reverse sync) remains unchanged.

Root cause from UAT: The `for events in rx` blocks on `rx.recv()` internally. After Ctrl+C sets `running=false`, no new events arrive, so the flag is never checked and the loop hangs.

<task type="auto" tdd="true">
  <name>Task 2: Fix reverse sync — watch parent directories instead of file paths</name>
  <files>crates/aisync-core/src/watch.rs</files>
  <behavior>
    - Test 1: tool_watch_paths returns parent directories of tool-native files, not the files themselves
    - Test 2: tool_watch_paths returns the set of expected filenames alongside the directories for event filtering
    - Test 3: is_tool_native detection works by checking if the event path filename matches an expected tool file in its parent directory
  </behavior>
  <action>
Refactor `tool_watch_paths` to return a struct or tuple that includes both the parent directories to watch AND the expected file paths for filtering:

1. **Change `tool_watch_paths` return type** to return `(Vec<PathBuf>, Vec<PathBuf>)` where the first vec is directories to watch and the second is expected file paths for event filtering. Or introduce a small struct `WatchTargets { watch_dirs: Vec<PathBuf>, expected_files: Vec<PathBuf> }`.

2. **In `tool_watch_paths`:** Instead of returning the file path directly, return the file's parent directory. Deduplicate directories (e.g., CLAUDE.md and AGENTS.md are both in project root). Still only include paths where the file exists and is NOT a symlink.

3. **Update the watcher setup** (lines 52-64): Watch the parent directories with `RecursiveMode::NonRecursive` instead of the file paths.

4. **Update `is_tool_native` detection** (lines 101-106): Instead of matching changed paths against `tool_paths` directly, check if any changed path's filename matches one of the expected tool filenames AND the changed path's parent matches one of the watched directories.

5. **Update existing tests** for the new return type:
   - `test_tool_watch_paths_returns_existing_non_symlink_files` — verify directories returned and expected files listed
   - `test_tool_watch_paths_skips_symlinked_files` — verify symlinked files excluded from expected_files
   - `test_tool_watch_paths_skips_missing_files` — verify empty when no files exist

Root cause from UAT: Editors (vim, VSCode, etc.) perform atomic saves by writing to a temp file then renaming. On macOS, kqueue watches are inode-based, so when the original file is replaced by rename, the watch is on the old (deleted) inode. Watching the parent directory avoids this because directory watches survive file replacements.
