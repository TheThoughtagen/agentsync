  <files>

  <action>
    **main.rs -- Add Hooks subcommand:**

    Add to the `Commands` enum:
    ```rust
    /// Manage hook definitions in .ai/hooks.toml
    Hooks {
        #[command(subcommand)]
        action: HooksAction,
    },
    ```

    Add `HooksAction` enum:
    ```rust
    #[derive(Subcommand)]
    enum HooksAction {
        /// List all hooks and per-tool support status
        List,
        /// Add a new hook definition interactively
        Add,
        /// Preview hook translations for all tools
        Translate,
    }
    ```

    Wire in main's match:
    ```rust
    Commands::Hooks { action } => commands::hooks::run_hooks(action, cli.verbose),
    ```

    **commands/hooks.rs -- Implement subcommands:**

    - **list**: Call `HookEngine::parse(project_root)` then `HookEngine::list_hooks()`. Display a table with columns: Event | Matcher | Command | Timeout. Below the hook table, show tool support status: "Claude Code: [supported] | OpenCode: [supported] | Cursor: [not supported]". Use colored output (green checkmark for supported, red X for not supported). If no hooks.toml found, print "No hooks defined. Use `aisync hooks add` to create one."

    - **add**: Interactive builder using `dialoguer`:
      1. `Select` prompt with VALID_EVENTS list for event type
      2. `Input` prompt for matcher (optional, allow empty string which becomes None)
      3. `Input` prompt for command (required)
      4. `Input` prompt for timeout in ms (optional, default empty)
      Then call `HookEngine::add_hook(project_root, event, matcher, command, timeout)`.
      For non-TTY: error with "aisync hooks add requires an interactive terminal" (per Phase 2 pattern).

    - **translate**: Call `HookEngine::parse(project_root)`. For each tool adapter, call `adapter.translate_hooks(&config)`. Display results with clear headers:
      ```
      === Claude Code (JSON) ===
      { "hooks": { ... } }

      === OpenCode (JS plugin stub) ===
      module.exports = { ... }

      === Cursor ===
      Warning: Cursor does not support hooks
      ```
      Use colored output: yellow for Cursor warning. Per CONTEXT.md: non-zero exit only on actual errors, not warnings.

    **commands/mod.rs -- Register module:**

    Add `pub mod hooks;`

    **sync.rs -- Integrate hooks into plan/execute:**

    In `SyncEngine::plan()`: After memory sync planning, check if `.ai/hooks.toml` exists. If so, parse with `HookEngine::parse()`, then for each enabled tool call `adapter.translate_hooks(&hooks_config)`. Based on the HookTranslation result:
    - `Supported { content, .. }`: Add `WriteHookTranslation { path, content, tool }` action to the tool's result. Path depends on tool:
      - Claude Code: `project_root.join(".claude/settings.json")` -- but NOTE: must merge into existing settings.json if it exists (read existing, add/update "hooks" key, write back). If no settings.json exists, create with just hooks.
      - OpenCode: `project_root.join(".opencode/plugins/aisync-hooks.js")`
    - `Unsupported { reason, .. }`: Add `WarnUnsupportedHooks { tool, reason }` action.

    In `SyncEngine::execute()`: Add handlers:
    - `WriteHookTranslation { path, content, tool }`: For Claude Code, read existing settings.json, parse as serde_json::Value, insert/update the "hooks" key with the translated content (parsed as Value), write back with `serde_json::to_string_pretty`. For OpenCode, create parent dirs and write JS file directly.
    - `WarnUnsupportedHooks { .. }`: No filesystem action, just recorded in results (display handled by CLI).
  </action>
