<objective>
Migrate ToolsConfig from named struct fields to BTreeMap<String, ToolConfig> with backward-compatible deserialization, enabling arbitrary tool names in aisync.toml.

Purpose: The current ToolsConfig has hardcoded fields (claude_code, cursor, opencode) that require code changes for each new tool. A BTreeMap with `#[serde(flatten)]` accepts any `[tools.X]` TOML section — making `[tools.windsurf]` or `[tools.codex]` work without config.rs changes. This completes the "single-file adapter" story.

Output: BTreeMap-based ToolsConfig, helper methods, refactored enabled_tools(), passing tests including round-trip and unknown-tool deserialization.
</objective>

1. **Refactor `enabled_tools()` in `crates/aisync-core/src/sync.rs`:**

Replace the hardcoded per-tool checks with a loop over `AnyAdapter::all_builtin()`:
```rust
pub(crate) fn enabled_tools(
    config: &AisyncConfig,
) -> Vec<(ToolKind, AnyAdapter, Option<&crate::config::ToolConfig>)> {
    let mut tools = Vec::new();
    for adapter in AnyAdapter::all_builtin() {
        let key = adapter.name().as_str().to_string();
        if config.tools.is_enabled(&key) {
            tools.push((
                adapter.name(),
                adapter,
                config.tools.get_tool(&key),
            ));
        }
    }
    tools
}
```

This eliminates the 3 hardcoded blocks (one per tool) and replaces them with a single loop.
